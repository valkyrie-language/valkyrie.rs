# 类型系统 (Type System)

Valkyrie 拥有一个高度表达且严谨的类型系统，旨在平衡开发效率、运行时性能与编译时安全性。

本章给出 Valkyrie 类型系统的主题划分与阅读顺序。

## 设计原则

Valkyrie 的类型系统基于以下原则：

1. **分层优于混层**: `row`、具名 `trait`、`class`、`unite`、`effect` 各自承担不同语义角色。
2. **组合优于误用继承**: 通过 ADT、row 与 trait 组合能力，而不是把所有抽象都塞进 class 层级。
3. **显式优于隐式**: 关键的类型转换、协议满足与效应追踪必须清晰可见。
4. **零成本抽象**: 所有的类型层级在单态化后均不产生运行时开销。
5. **代数一致性**: 类型之间的关系（子类型、交并集）遵循严格的数学格论。

## 形式化约定

本章后续文档在需要精确定义语义边界时，统一采用以下约定：

- `A <: B` 表示 `A` 是 `B` 的子类型；该记号只在文档明确说明的子类型关系中使用。
- `A ⊨ T` 表示类型 `A` 满足具名 `trait T`，并可形成稳定的协议事实。
- `M(A)` 表示类型 `A` 对外暴露的方法面；`field` 在语义上视为 `getter + setter`，因此最终也计入 `M(A)`。
- `M(A) ⊒ R` 表示 `A` 的方法面覆盖匿名 row requirement `R`。
- 文中使用“必须 / 不得 / 应 / 可以”时，分别对应强制规则、禁止规则、推荐规则与允许规则。

## 基本分层

本章统一采用下列语义分层：

- `row`：匿名方法约束，只回答“你会不会这些方法”。
- `trait`：具名协议，只回答“你是否满足这个被命名的协议”。
- `class`：对象类型，只回答“你是不是这个类层级里的成员”。
- `unite`：联合类型，只回答“这个值当前属于哪一个分支”。
- `effect`：handler 与能力边界，只回答“你在什么 effect 环境里调用它”。

在这组分层里：

- `class / sealed class / unite` 的匹配按声明身份处理。
- 具名 `trait` 的匹配按协议满足处理。
- 匿名 row 的匹配按方法面覆盖处理。
- `effect` 的匹配按 effect 集与 handler 证据处理。
