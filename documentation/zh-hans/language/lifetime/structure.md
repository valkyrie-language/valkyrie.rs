# 值类型 (Structure)

`structure` 是 Valkyrie 构建高性能系统的基石，代表了严谨的**值语义 (Value Semantics)**。它在内存中采用内联存储，遵循严格的移动语义，是实现确定性资源管理与硬件级性能优化的核心。

## 内存布局：内联与局部性

`structure` 默认分配在栈上，其最大的物理特征是**内联 (Inline) 存储**。当一个结构体作为另一个结构的字段时，它会直接嵌入父结构的内存布局中，完全消除了额外的指针跳转开销。

```valkyrie
structure Vec2 { x: f32, y: f32 }

structure Rect {
    origin: Vec2, # 直接内联在 Rect 内存块中
    size: Vec2,   # 直接内联在 Rect 内存块中
}
```

这种布局方案对 CPU 缓存 (Cache) 极其友好。通过保证数据的**空间局部性**，编译器能够大幅提升高速缓存的命中率，从而压榨出硬件的极致性能。

## 所有权与权属流转

`structure` 的核心运行逻辑是**移动语义 (Move Semantics)**。不同的权属控制符决定了底层数据的流转方式：

### 1. 默认与 `own`：所有权彻底转移

当你将 `structure` 赋值给新变量或以 `own` 模式传递时，底层数据会发生**物理位拷贝 (Bitwise Copy)**，同时原变量在逻辑上被立即标记为失效（不可访问）。

#### 物理拷贝的必要性
虽然逻辑上同一时刻只有一个所有者，但在物理实现中，`structure` 的位置是固定的（位于特定的栈 Slot）。当权属转移到新变量或跨越栈帧时，数据必须被搬运到新的物理地址以维持内联特性。

- **逻辑语义**：所有权转移 (Move)，原变量生命周期终结。
- **物理效应**：执行高效的 `memcpy` 将数据搬运至新 Slot。

```valkyrie
let v1 = Vec2 { x: 1.0, y: 1.0 }
let v2 = v1 # 逻辑移动，物理搬运；此后 v1 不再可用
```

### 2. `ref` 与 `mut`：零拷贝借用
借用操作不会触动所有权。在底层实现中，编译器仅传递该结构的物理内存地址（指针）。

```valkyrie
# 仅传递地址，不涉及任何数据搬运
micro distance(ref a: Vec2, ref b: Vec2) -> f32 { ... }
```

### 3. 显式复制：`.copy()` 与 `.clone()`
若需保留原对象并产生独立副本，必须通过显式调用表达：
- **`.copy()`**：执行简单的位拷贝。仅适用于不持有复杂外部资源（如原始指针、句柄）的“平凡”结构。
- **`.clone()`**：触发用户定义的深拷贝逻辑，用于处理包含复杂权属关系的资源复制。

## AIFD 编排：自定义行为

开发者可以通过实现特定的生命周期函数，对 `structure` 的关键阶段实施精准干预：

- **`initiate(mut self)`**：在 A 阶段（分配）之后立即触发。常用于确立初始不变式或验证状态。
- **`finalize(mut self)`**：在 D 阶段（释放）之前触发。对于 `structure`，这是**物理确定性**触发的（即离开作用域时立即执行）。

```valkyrie
structure TempFile {
    path: utf8,
    
    # 利用 RAII 确保资源的确定性清理
    finalize(mut self) {
        # 离开作用域时，确保物理文件被从磁盘移除
        std::fs::remove(self.path)
    }
}
```

## 硬件级物理优化 (Black Magic)

为了达成极致的运行效率，Valkyrie 编译器在严格维持逻辑语义的前提下，会对 `structure` 施加多种物理层面的深度优化：

### 1. 移动消除 (Move Elision)
虽然 `move` 在逻辑上涉及拷贝，但编译器会通过以下技术消除 90% 以上的实际开销：
- **原地构造 (In-place Construction)**：通过 **NRVO (Named Return Value Optimization)**，编译器直接在目标变量的物理位置上生成对象，完全跳过临时变量的创建与搬运。
- **寄存器映射**：对于小型结构（如 `Vec2`），它们通常被直接分配到 CPU 寄存器。此时的“移动”仅是逻辑上的重命名，不产生任何内存 I/O。
- **尾调用转发**：当变量作为最后一次使用传递给下游函数时，编译器会尝试让新函数复用当前的内存 Slot。

### 2. 布局重排 (Field Reordering)
为了对齐内存并减小体积，编译器会自动优化字段的物理顺序：
- **消除空隙 (Padding)**：通过按字段大小降序排列，挤压掉为了内存对齐而产生的填充空间。
- **空间压榨**：原本占用 24 字节的非对齐结构，优化后可能仅需 16 字节，显著提升缓存行 (Cache Line) 的利用率。

### 3. 标量替换 (SROA)
若结构体仅在局部使用且未发生逃逸，编译器会将其“拆解”为独立的标量变量：
- **寄存器化**：结构体的各个字段被分别映射到不同的 CPU 寄存器，完全消除内存访问。
- **解除耦合**：一旦拆解，死代码消除等传统优化即可针对单个字段生效。

### 4. 零大小类型 (ZST)
不含任何字段的 `structure` 在运行时不占用任何物理空间 (0 Byte)。
- **零成本抽象**：你可以定义海量的 ZST 来强化类型安全或表达状态机逻辑，而不会对生成的机器码产生任何性能负担。

### 5. 透明包装 (Transparent)
单字段包装类（如 `UserId(u64)`）在 ABI 层面与底层类型完全等价。
- **无感包装**：在 FFI 调用中直接映射为原始类型，实现零开销的语义强化。

### 6. 空位优化 (Niche Optimization)
编译器会利用字段中非法位模式（如非空指针的 0 值）来存储 `Option` 的标签。
- **内存减负**：使得 `Option<ref T>` 的大小与 `ref T` 保持完全一致，实现零空间损耗的错误处理。

### 7. SIMD 自动矢量化
对于符合对齐要求的数值结构（如 `Vec4`），编译器会利用 SSE/AVX 指令集实现单指令多数据并行：
- **吞吐量激增**：一次性处理 4 个或更多浮点运算，特别适用于图形学、物理引擎及高性能科学计算场景。

## 最佳实践：核心决策建议

- **默认首选 `structure`**：适用于数学向量、配置快照、小型状态机以及所有追求极致性能的轻量级数据模型。
- **拥抱内联**：通过合理组合 `structure` 减少不必要的堆分配，构建缓存友好的数据结构。
- **审视移动成本**：对于体积巨大的结构（如包含超大固定数组），频繁的 `own` 传递可能带来物理拷贝开销，此时建议优先考虑使用 `ref` 借用。
