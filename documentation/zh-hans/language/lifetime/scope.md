# 作用域与自动生命周期插入 (Scope & Auto-insertion)

Valkyrie 摒弃了晦涩且繁琐的显式生命周期标注，转而通过**词法作用域 (Lexical Scope)**、**参数控制符 (Parameter Qualifiers)** 以及深度的**静态控制流分析 (Static CFG Analysis)** 来实现 AIFD 全生命周期的自动化管理。

## 判定核心：词法作用域

在 Valkyrie 中，大括号 `{}` 定义了一个严密的物理边界与执行时间线。编译器以此为基准，精确编排资源的诞生与消亡。

### 1. 分配与初始化 (Allocate & Initiate)
当变量在作用域内被绑定时，编译器会立即插入 A 和 I 阶段的代码。
- **值类型 (`structure`)**：通常表现为栈指针 (Stack Pointer) 的移动，实现近乎零成本的分配。
- **引用类型 (`class`)**：触发受管堆内存的分配请求，并执行构造逻辑。

### 2. 终结与释放 (Finalize & Delocate)
编译器会在作用域结束处，或所有可能的提前退出路径（如 `return`、`break`、`raise`）上，自动注入必要的销毁逻辑。

```valkyrie
{
    let p = Point { x: 1.0, y: 2.0 }
    if some_condition {
        # 编译器在此处自动插入 p.finalize() 并安排内存回收
        return
    }
}
# 作用域正常退出时，编译器同样会自动确保销毁代码的执行
```

## 权属管理：控制符与默认策略

Valkyrie 引入了控制符（`ref`、`mut`、`own`）来精确定义函数对参数的权属。为了平衡性能与开发体验，编译器为不同类型量身定制了直观的默认策略。

### 1. 显式控制符语义

- **引用模式 (`ref` / `mut`)**：代表**借用 (Borrowing)**。不发生所有权转移。原始作用域在函数调用结束后，依然保留对该对象的销毁责任。
- **所有权模式 (`own`)**：代表**接管 (Taking Ownership)**。发生**移动 (Move)**。对象的生命周期终结责任从调用方彻底转移到被调用方。

### 2. 隐式权属策略 (Implicit Policy)

| 类型分类 | 默认控制符 | 核心设计初衷 |
| :--- | :--- | :--- |
| **Structure** (值类型) | **`own`** | 确保值语义的独立性，彻底杜绝隐蔽的跨作用域副作用。 |
| **Class** (引用类型) | **`ref`** | 提供透明且自然的共享体验，符合托管对象的编程直觉。 |

```valkyrie
micro process(d: Data)   # 隐式等价于 (own d: Data)
micro display(p: Person) # 隐式等价于 (ref p: Person)
```

### 3. 设计深度：根除“隐式共享可变性”陷阱

Valkyrie 的默认策略旨在从根本上消除 Python、JavaScript 等动态语言中常见的权属模糊隐患。

#### 场景 A：共享可变状态导致的副作用
在传统托管语言中，简单的赋值往往会引入危险的隐式共享：
```python
# Python 示例：隐式共享导致数据被意外修改
a = [1, 2]
b = a
a.append(3) 
print(b) # 输出 [1, 2, 3] —— b 的内容在开发者未察觉的情况下改变了
```

**Valkyrie 的防御机制**：
1. **针对 Class (引用类型)**：默认采用 **`ref` (只读引用)**，严格遵循 **读写互斥原则**。
    ```valkyrie
    let mut a = [1, 2]
    let b = a        # b 自动获得 ref [] 权限
    a.push(3)        # 编译错误！无法修改 'a'，因为 'b' 正在以只读模式借用它
    ```
2. **针对 Structure (值类型)**：默认采用 **`own` (所有权移动)**。
    ```valkyrie
    let a = Point { x: 1, y: 2 }
    let b = a        # a 的所有权已完整移动至 b，a 在当前作用域失效
    print(a.x)       # 编译错误！'a' 已被标记为失效，无法再次访问
    ```

#### 场景 B：可变默认参数的状态残留
另一个经典的陷阱是函数默认参数在多次调用间的状态持久化：
```python
# Python 示例：默认参数变成了全局状态
def add_to(item, list=[]):
    list.append(item)
    return list

print(add_to(1)) # [1]
print(add_to(2)) # [1, 2] —— 默认列表竟然在调用间被共享了！
```

**Valkyrie 的防御机制**：坚持 **默认不可变性** 与 **显式权限授权**。
1. **参数默认只读**：无论是 `class` 还是 `structure`，作为参数传递时默认均为只读。
    ```valkyrie
    micro add_to(item: i32, list: [i32] = []) {
        list.push(item) # 编译错误！'list' 默认仅具只读权限
    }
    ```
2. **`mut` 严禁携带默认值**：若函数确需修改传入状态，必须显式声明 `mut`，且编译器强制禁止为此类参数设置默认值。
    ```valkyrie
    # 推荐写法：调用者能一眼识别潜在的副作用
    micro update(mut data: Data) { ... }

    # 编译错误！带有 'mut' 标识的参数严禁拥有默认值，必须由调用方显式传入
    micro wrong(mut list: [i32] = []) { ... }
    ```

**设计哲学总结**：可变状态的“所有者”必须始终清晰。Valkyrie 强制要求：**凡涉及状态修改的操作，必须由调用方通过显式权限进行授权，确保整个权属链条的透明、可追溯与完整。**

---

## 核心决策指南

- **仅需只读访问？** 请使用默认的 `ref`（引用类型）或显式 `ref`（值类型）。
- **需要原地修改？** 请显式声明 `mut` 控制符。
- **需要接管对象或移入容器？** 对于值类型请使用默认行为，对于引用类型请使用 `own`。
- **快速记忆**：值类型 (`structure`) 默认移动，引用类型 (`class`) 默认共享。

## 深度静态分析：超越词法边界

当执行流跨越简单的词法边界时，编译器会通过高阶静态分析确保生命周期管理的严密性。

### 1. 控制流感知 (CFG) 与线性类型追踪 (Linear Types)

Valkyrie 编译器通过构建**控制流图 (Control Flow Graph)**，确保每个拥有所有权的对象在任何可能的执行路径上都必须被“消费”且仅能被消费一次。

#### 确定性回滚 (Deterministic Unwinding)
在涉及 `?` 错误传播、提前 `return` 或 `raise` 的复杂逻辑中，编译器会实时计算当前执行点的"存活变量集合 (Live Set)"，并严格按照与初始化相反的顺序自动插入销毁代码。

```valkyrie
micro process_file() -> Result {
    let f1 = File::open("a.txt")? # 若失败，无事发生
    let f2 = File::open("b.txt")? # 若失败，自动插入 f1.finalize()
    let res = compute(f1, f2)?    # 若失败，自动插入 f2.finalize(), f1.finalize()
    Success
}
```

#### 分支汇合与权属补偿机制
在 `if` 或 `match` 分支中，若不同路径导致的所有权状态不一致，编译器会在汇合点之前进行自动“补偿”以对齐状态：

```valkyrie
let x = Data::new()
if condition {
    consume(x) # 路径 A: x 的所有权被移动至 consume 函数
} else {
    # 路径 B: x 仍然存活
    # 编译器在此处自动插入 x.finalize()
    # 确保汇合点之后，x 在所有路径上均已失效
}
# 汇合点：x 状态达成一致（均为失效），后续严禁访问
```

### 2. 逃逸分析 (Escape Analysis)

逃逸分析不仅决定了销毁的位置，还直接指导了后端的物理分配策略。

#### 逃逸路径判定
若对象具备以下特征之一，则标记为“逃逸 (Escaped)”，其生命周期将超越当前作用域：
1. **持久化存储**：被存入全局变量、单例或长生命周期的容器中。
2. **异步捕获**：被 `spawn` 或 `async` 闭包捕获，进入并行的后台执行流。
3. **权属返回**：作为函数返回值被 `own` 传回给调用方。
4. **类型擦除**：被转换为 `any` 动态类型或通过 FFI 接口传递给外部语言。

#### 物理优化反馈
- **栈提升 (Stack Promotion)**：若一个 `class` 对象被判定为未逃逸，编译器会将其从堆分配提升为栈分配。这不仅规避了 GC 压力，还通过 CPU 缓存局部性带来了巨大的性能提升。
- **标量替换 (SROA)**：若 `structure` 未逃逸且仅局部访问，编译器会将其拆解为独立的标量变量并映射到寄存器，完全消除内存访问。

### 3. 非局部跳转：Effect 系统与作用域管理

Valkyrie 的代数效应 (Algebraic Effects) 支持灵活的执行流挂起与恢复，这对生命周期管理提出了极高要求。

#### 挂起 (Suspension)
当代码执行 `raise` 操作触发效应时，当前的执行流和变量环境会被挂起。此时，作用域内的变量**不会触发** `finalize`，因为它们可能在未来被 Handler 恢复执行。

#### 截断清理 (Truncation Cleanup)
如果 Handler 决定终止当前的执行流（不再 `resume`），编译器将识别所有相关的挂起作用域，并按照栈回溯顺序强制执行所有存活变量的清理逻辑，确保资源不泄露。

#### 作用域快照
对于需要多次恢复执行的效应（如非确定性计算），编译器会透明地为作用域状态生成快照，并在每次 `resume` 时恢复 AIFD 模型的中间状态。
