# 外部对象 (Foreign Objects)

在 Valkyrie 与 C、C++ 或 Rust 等外部语言进行深度交互 (FFI) 时，严密且精准地管理外部对象的生命周期是确保系统稳定性、防止内存泄漏以及维持高性能执行的关键。

## FFI 与 AIFD 模型的深度映射

由于外部对象通常由其宿主语言（如 C 堆或 Rust 分配器）管理，不受 Valkyrie 垃圾回收 (GC) 机制的直接控制，开发者必须手动将其生命周期显式映射到 AIFD 模型中。

特别是 **F (Finalize, 终结)** 阶段的映射至关重要。这确保了当 Valkyrie 侧的包装对象失效时，底层的外部资源（如原始指针、系统句柄、网络套接字等）能够被确定性地释放。

### 实战示例：安全封装 C++ 对象

```valkyrie
# 导入外部 C++ 原生接口
@import(c, "lib", "obj_new")
micro obj_new() -> ◆u8

@import(c, "lib", "obj_delete")
micro obj_delete(ptr: ◆u8)

# Valkyrie 安全包装类
class ExternalWrapper {
    ptr: ◆u8
    
    # 状态初始化 (I)：在构造时申请外部资源
    initiate(mut self) {
        unsafe { 
            self.ptr = obj_new() 
        }
    }
    
    # 逻辑终结 (F)：确保在对象销毁前，外部资源被安全释放
    finalize(mut self) {
        if self.ptr != null {
            unsafe { 
                obj_delete(self.ptr) 
                self.ptr = null # 防御性编程：防止双重释放
            }
        }
    }
}
```

## 安全包装模式 (Safe Guards)

为了提供更卓越的开发体验 (UX) 并降低出错风险，建议采用以下模式利用 `Finalize` 自动管理外部资源：

1.  **对象句柄模式 (Handle Pattern)**：将外部原始指针封装在 Valkyrie 的 `class` 中。这样，外部资源的存续就与 Valkyrie 对象的生命周期紧密绑定，由 GC 间接驱动清理逻辑。
2.  **RAII 守护者 (RAII Guardians)**：通过 `finalize` 方法自动触发外部语言的析构函数或资源回收例程，实现类似于 Rust RAII 的确定性、自动化的清理机制。

## 核心内存安全准则

- **空指针防御 (Null-Safety)**：在 `finalize` 的实现中，必须始终校验外部指针的有效性。这不仅能防止双重释放 (Double Free)，还能增强程序在异常情况下的健壮性。
- **内存权属与固定 (Pinning)**：若需要将 Valkyrie 托管的内存地址（如数组缓冲区）传递给外部语言异步使用，必须确保该内存在外部调用期间不会被 GC 移动。此时应使用 `Pin` 机制锁定对象位置。

## 外部对象的底层物理优化

为了让跨语言调用达到原生级别的执行效率，Valkyrie 编译器对 FFI 交互路径进行了深度的针对性优化：

### 1. 零拷贝数据交换 (Zero-copy Exchange)
Valkyrie 的 `structure` 类型在内存布局上与标准 C 严格兼容。
- **指针透传**：在传递复杂结构体时，编译器直接传递原始内存指针，彻底消除昂贵的数据拷贝过程。
- **常数级开销**：即便处理包含数百万个元素的巨型顶点缓冲区，跨语言传递的性能损耗也始终维持在 $O(1)$ 级别。

### 2. 极速调用路径 (Fast-path FFI)
编译器根据目标平台的 ABI（如 System V 或 Windows x64）精准编排调用序列。
- **寄存器直传**：轻量级外部对象（如资源句柄、短向量等）直接通过 CPU 寄存器传递，完全绕过慢速的栈内存。
- **内建指令替换**：针对数学计算等标准库热点（如 `sin`, `sqrt`），编译器会尝试将其直接替换为等效的硬件 CPU 指令。

### 3. 终结器批量调度 (Finalizer Batching)
释放外部对象通常涉及跨语言上下文切换的昂贵成本。
- **延迟合并策略**：GC 会智能汇总多个待执行的外部终结逻辑，并在单次扫描周期内进行批量触发。
- **流水线优化**：这种方式极大地减少了 CPU 在 Valkyrie 运行时与外部运行时之间频繁切换产生的流水线冲刷 (Pipeline Flush) 损耗。

### 4. 包装器剥离与内联
- **局部性优化**：若外部包装器实例未逃逸出当前函数作用域，编译器会尝试剥离包装层，直接操作原始指针，并内联所有的清理代码。
- **栈提升**：包装指针的 `class` 实例本身可能会被优化至栈上分配，从而完全消除堆分配带来的开销。

### 5. 跨语言全程序优化 (LTO)
当外部库以静态链接方式整合时，Valkyrie 支持开启跨语言的全程序分析：
- **边界内联 (Boundary Inlining)**：编译器可以打破语言壁垒，将外部库中的热点代码直接内联到 Valkyrie 的调用处。
- **死代码剔除 (DCE)**：在链接阶段精准识别并移除外部库中未被调用的冗余逻辑，极大地压缩二进制体积。
