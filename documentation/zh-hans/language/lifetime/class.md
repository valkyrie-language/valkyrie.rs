# 引用类型 (Class)

Valkyrie 默认采用最先进的垃圾回收 (GC) 机制进行内存管理。对于绝大多数应用层代码而言，GC 提供了零负担的内存安全保障、极佳的开发体验以及高度自动化的资源回收能力。

## 类型声明：`class`

在 Valkyrie 中，通过 `class` 关键字定义的类型均为**引用类型**。这意味着变量存储的是指向堆上实例的引用，而非对象数据本身。

```valkyrie
class Person {
    name: utf8,
    age: i32,
}

# 实例化：默认在受管堆 (Managed Heap) 上进行分配
let p1 = Person { name: "Valkyrie", age: 1 }
```

### 核心特性

1.  **全自动生命周期管理**：对象的完整生命轨迹由高性能 GC 引擎自动追踪，开发者可以完全专注于业务逻辑，无需担心内存释放。
2.  **默认引用共享语义**：在对 `class` 实例进行赋值或参数传递时，系统默认采用 **`ref`** 模式共享引用（即传递指针）。这种设计从根源上规避了传统语言中常见的"隐式共享可变性"陷阱。
3.  **循环引用自动解环**：GC 能够精准识别并安全回收存在复杂循环引用关系的对象集群，确保系统在处理大规模关联数据时依然具备卓越的稳定性。

```valkyrie
let p1 = Person { name: "Valkyrie", age: 1 }
let p2 = p1  # 自动以 ref 模式共享引用，此时 p1 和 p2 指向同一个物理内存实例
```

## 权属模型：共享与内部可变性

在 AIFD 模型下，`class` 默认采用 **引用共享 (Shared)** 语义。为了确保在这种共享模式下的内存安全，Valkyrie 引入了以下机制：

### 1. 默认不可变性与 `ref`
在对 `class` 实例进行赋值或参数传递时，默认采用 `ref` 模式。这意味着多个变量可以同时指向同一个实例，但它们默认仅具有只读权限。

### 2. 响应式容器 (Signal)
针对需要跨组件同步状态的场景，Valkyrie 提倡使用 **[Signal](../reactive-programming/signal.md)**。它将数据封装在具备观察者模式的容器内，通过显式的响应式绑定而非隐蔽的指针操作来管理状态流转。

### 3. 内部可变性 (Cell/Atom)
对于特定场景，可以使用 `Cell`（单线程安全）或 `Atom`（跨线程安全）在持有只读引用的情况下修改内部数据。

**核心哲学：默认安全、显式共享。**

## GC 与 AIFD 模型的协同演进

对于受管对象，Valkyrie 运行时通过一套严密的机制自动编排 AIFD 生命周期阶段：

- **Allocate (分配)**: GC 从受管堆空间或专用的**线程本地分配缓冲区 (TLAB)** 中秒级划拨内存。
- **Initiate (初始化)**: 编译器自动注入指令，在分配成功的内存上立即执行对象的 `initiate` 方法。
- **Finalize (终结)**: 当 GC 引擎通过可达性分析确认对象不再被使用时，会调度并异步执行其 `finalize` 清理逻辑。
- **Delocate (释放)**: 在所有终结操作圆满完成后，GC 正式将对应的物理内存块回收并标记为可用。

### Alloy：权属系统与 GC 的完美交织

Valkyrie 的类型系统深度融合了 **Alloy** 研究论文 (*Garbage Collection for Rust: The Finalizer Frontier*) 的前沿成果，实现了静态权属检查与动态垃圾回收的无缝整合。

1.  **静态终结分析 (Static Finalization Analysis)**：编译器会深度剖析 `Finalize` 的实现细节。若确定终结逻辑不产生外部副作用（纯内存操作），GC 即可采用并行化或延迟执行策略，从而显著降低系统的停顿时间 (Stop-the-world)。
2.  **权属集成架构**：GC 并非所有权的替代方案，而是其强力补充。受管对象可以安全地持有手动管理对象的引用，反之亦然。编译器通过严密的生命周期契约确保了此类跨界引用的绝对安全性。
3.  **确定性资源回收**：尽管内存回收是异步的，但对于实现了 `Finalize` 的关键系统资源（如网络 Socket、文件句柄），Valkyrie 会在对象失效后以最高优先级触发其终结逻辑。

## 引用类型与值类型深度对比

| 核心维度 | class (引用类型) | structure (值类型) |
| :--- | :--- | :--- |
| **内存物理布局** | 默认在受管堆分配，通过指针访问 | 默认在栈上分配或在父结构中内联存储 |
| **默认赋值语义** | **引用共享 (Shared)** | **所有权移动 (Move)** |
| **克隆/复制行为** | 无需显式复制，共享同一实例 | 必须显式调用 `.copy()` 或 `.clone()` |
| **生命周期驱动** | 由后台 GC 引擎自动追踪与回收 | 遵循严格的词法作用域规则 (RAII) |
| **典型适用场景** | 具有唯一身份的业务实体、复杂状态机 | 轻量级数据容器、数学向量、性能敏感组件 |

## 引用类型的底层物理优化

尽管 `class` 提供了高度抽象的编程模型，但 Valkyrie 编译器通过一系列先进的优化技术，致力于将运行时开销降至最低：

### 1. 逃逸分析 (Escape Analysis)
编译器会全程追踪对象的生命周期。若能静态证明某个 `class` 实例不会逃逸出当前函数，它会将其**从堆分配自动优化为栈分配**，甚至直接拆解并映射到 CPU 寄存器，从而完全消除 GC 压力。

### 2. 深度去虚化 (Devirtualization)
利用上下文感知与类型流分析，编译器能将原本昂贵的动态派发（虚函数查找）重写为直接的函数调用。这不仅消除了跳转开销，还为后续的**函数内联 (Inlining)** 开启了巨大的空间。

### 3. 指针压缩 (Pointer Compression)
在 64 位环境下，若堆内存需求在 32GB 以内，编译器将使用 32 位压缩指针替代 64 位绝对地址。这能显著缩减对象在内存中的体积，并成倍提升 CPU 缓存 (L1/L2 Cache) 的利用率。

### 4. 线程本地分配 (TLAB)
每个执行线程都拥有私有的快速分配缓冲区。在实例化对象时，仅需简单的指针偏移 (Pointer Bumping) 即可完成，无需任何全局锁竞争，其速度几乎等同于原始的栈操作。

### 5. 静态终结捷径
若 `finalize` 方法被判定为“平凡终结”（无外部副作用），GC 在扫描回收阶段会直接略过终结队列，大幅缩短垃圾回收周期。

### 6. 写屏障消除 (Barrier Elimination)
在分代式 GC 架构中，编译器能够智能识别并移除冗余的写屏障指令（例如对新创建对象的初始属性赋值），进一步压低维护对象引用关系的成本。
