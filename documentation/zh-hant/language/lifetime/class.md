# 引用型別 (Class)

Valkyrie 預設採用最先進的垃圾回收 (GC) 機制進行記憶體管理。對於絕大多數應用層代碼而言，GC 提供了零負擔的記憶體安全保障、極佳的開發體驗以及高度自動化的資源回收能力。

## 型別宣告：`class`

在 Valkyrie 中，透過 `class` 關鍵字定義的型別均為**引用型別**。這意味著變數儲存的是指向堆上實例的引用，而非物件資料本身。

```valkyrie
class Person {
    name: utf8,
    age: i32,
}

# 實例化：預設在受管堆 (Managed Heap) 上進行分配
let p1 = Person { name: "Valkyrie", age: 1 }
```

### 核心特性

1.  **全自動生命週期管理**：物件的完整生命軌跡由高效能 GC 引擎自動追蹤，開發者可以完全專注於業務邏輯，無需擔心記憶體釋放。
2.  **預設引用共享語義**：在對 `class` 實例進行賦值或參數傳遞時，系統預設採用 **`ref`** 模式共享引用（即傳遞指標）。這種設計從根源上規避了傳統語言中常見的「隱式共享可變性」陷阱。
3.  **循環引用自動解環**：GC 能夠精準識別並安全回收存在複雜循環引用關係的物件集群，確保系統在處理大規模關聯資料時依然具備卓越的穩定性。

```valkyrie
let p1 = Person { name: "Valkyrie", age: 1 }
let p2 = p1  # 自動以 ref 模式共享引用，此時 p1 和 p2 指向同一個物理記憶體實例
```

## 權屬模型：共享與內部可變性

在 AIFD 模型下，`class` 預設採用 **引用共享 (Shared)** 語義。為了確保在這種共享模式下的記憶體安全，Valkyrie 引入了以下機制：

### 1. 預設不可變性與 `ref`
在對 `class` 實例進行賦值或參數傳遞時，預設採用 `ref` 模式。這意味著多個變數可以同時指向同一個實例，但它們預設僅具有唯讀權限。

### 2. 響應式容器 (Signal)
針對需要跨元件同步狀態的場景，Valkyrie 提倡使用 **[Signal](../reactive-programming/signal.md)**。它將資料封裝在具備觀察者模式的容器內，透過顯式的響應式綁定而非隱蔽的指標操作來管理狀態流轉。

### 3. 內部可變性 (Cell/Atom)
對於特定場景，可以使用 `Cell`（單執行緒安全）或 `Atom`（跨執行緒安全）在持有唯讀引用的情況下修改內部資料。

**核心哲學：預設安全、顯式共享。**

## GC 與 AIFD 模型的協同演進

對於受管物件，Valkyrie 執行階段透過一套嚴密的機制自動編排 AIFD 生命週期階段：

- **Allocate (分配)**: GC 從受管堆空間或專用的**執行緒本地分配緩衝區 (TLAB)** 中秒級劃撥記憶體。
- **Initiate (初始化)**: 編譯器自動插入指令，在分配成功的記憶體上立即執行物件的 `initiate` 方法。
- **Finalize (終結)**: 當 GC 引擎透過可達性分析確認物件不再被使用時，會調度並非同步執行其 `finalize` 清理邏輯。
- **Delocate (釋放)**: 在所有終結操作圓滿完成後，GC 正式將對應的物理記憶體塊回收並標記為可用。

### Alloy：權屬系統與 GC 的完美交織

Valkyrie 的型別系統深度融合了 **Alloy** 研究論文 (*Garbage Collection for Rust: The Finalizer Frontier*) 的前沿成果，實現了靜態權屬檢查與動態垃圾回收的無縫整合。

1.  **靜態終結分析 (Static Finalization Analysis)**：編譯器會深度剖析 `Finalize` 的實作細節。若確定終結邏輯不產生外部副作用（純記憶體操作），GC 即可採用並行化或延遲執行策略，從而顯著降低系統的停頓時間 (Stop-the-world)。
2.  **權屬整合架構**：GC 並非所有權的替代方案，而是其強力補充。受管物件可以安全地持有手動管理物件的引用，反之亦然。編譯器透過嚴密的生命週期契約確保了此類跨界引用的絕對安全性。
3.  **確定性資源回收**：儘管記憶體回收是非同步的，但對於實作了 `Finalize` 的關鍵系統資源（如網路 Socket、檔案控制代碼），Valkyrie 會在物件失效後以最高優先級觸發其終結邏輯。

## 引用型別與值型別深度對比

| 核心維度 | class (引用型別) | structure (值型別) |
| :--- | :--- | :--- |
| **記憶體物理佈局** | 預設在受管堆分配，透過指標存取 | 預設在堆疊上分配或在父結構中內嵌儲存 |
| **預設賦值語義** | **引用共享 (Shared)** | **所有權移動 (Move)** |
| **克隆/複製行為** | 無需顯式複製，共享同一實例 | 必須顯式呼叫 `.copy()` 或 `.clone()` |
| **生命週期驅動** | 由後台 GC 引擎自動追蹤與回收 | 遵循嚴格的詞法作用域規則 (RAII) |
| **典型適用場景** | 具有唯一身份的業務實體、複雜狀態機 | 輕量級資料容器、數學向量、效能敏感元件 |

## 引用型別的底層物理優化

儘管 `class` 提供了高度抽象的編程模型，但 Valkyrie 編譯器透過一系列先進的優化技術，致力於將執行階段開銷降至最低：

### 1. 逃逸分析 (Escape Analysis)
編譯器會全程追蹤物件的生命週期。若能靜態證明某個 `class` 實例不會逃逸出當前函式，它會將其**從堆分配自動優化為堆疊分配**，甚至直接拆解並映射到 CPU 暫存器，從而完全消除 GC 壓力。

### 2. 深度去虛擬化 (Devirtualization)
利用上下文感知與型別流分析，編譯器能將原本昂貴的動態派發（虛擬函式查找）重寫為直接的函式呼叫。這不僅消除了跳轉開銷，還為後續的**函式內嵌 (Inlining)** 開啟了巨大的空間。

### 3. 指標壓縮 (Pointer Compression)
在 64 位元環境下，若堆記憶體需求在 32GB 以內，編譯器將使用 32 位元壓縮指標替代 64 位元絕對位址。這能顯著縮減物件在記憶體中的體積，並成倍提升 CPU 快取 (L1/L2 Cache) 的利用率。

### 4. 執行緒本地分配 (TLAB)
每個執行執行緒都擁有私有的快速分配緩衝區。在實例化物件時，僅需簡單的指標偏移 (Pointer Bumping) 即可完成，無需任何全域鎖競爭，其速度幾乎等同於原始的堆疊操作。

### 5. 靜態終結捷徑
若 `finalize` 方法被判定為「平凡終結」（無外部副作用），GC 在掃描回收階段會直接略過終結隊列，大幅縮短垃圾回收週期。

### 6. 寫屏障消除 (Barrier Elimination)
在分代式 GC 架構中，編譯器能夠智慧識別並移除多餘的寫屏障指令（例如對新建立物件的初始屬性賦值），進一步壓低維護物件引用關係的成本。
