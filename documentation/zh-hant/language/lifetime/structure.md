# 值型別 (Structure)

`structure` 是 Valkyrie 構建高效能系統的基石，代表了嚴謹的**值語義 (Value Semantics)**。它在記憶體中採用內聯儲存，遵循嚴格的移動語義，是實作確定性資源管理與硬體級效能優化的核心。

## 記憶體佈局：內聯與局部性

`structure` 預設分配在堆疊上，其最大的物理特徵是**內聯 (Inline) 儲存**。當一個結構體作為另一個結構的欄位時，它會直接嵌入父結構的記憶體佈局中，完全消除了額外的指標跳轉開銷。

```valkyrie
structure Vec2 { x: f32, y: f32 }

structure Rect {
    origin: Vec2, # 直接內聯在 Rect 記憶體塊中
    size: Vec2,   # 直接內聯在 Rect 記憶體塊中
}
```

這種佈局方案對 CPU 快取 (Cache) 極其友好。透過保證資料的**空間局部性**，編譯器能夠大幅提升快取的命中率，從而壓榨出硬體的極致效能。

## 所有權與權屬流轉

`structure` 的核心執行邏輯是**移動語義 (Move Semantics)**。不同的權屬控制符決定了底層資料的流轉方式：

### 1. 預設與 `own`：所有權徹底轉移

當你將 `structure` 賦值給新變數或以 `own` 模式傳遞時，底層資料會發生**物理位拷貝 (Bitwise Copy)**，同時原變數在邏輯上被立即標記為失效（不可訪問）。

#### 物理拷貝的必要性
雖然邏輯上同一時刻只有一個所有者，但在物理實作中，`structure` 的位置是固定的（位於特定的堆疊 Slot）。當權屬轉移到新變數或跨越堆疊幀時，資料必須被搬運到新的物理地址以維持內聯特性。

- **邏輯語義**：所有權轉移 (Move)，原變數生命週期終結。
- **物理效應**：執行高效的 `memcpy` 將資料搬運至新 Slot。

```valkyrie
let v1 = Vec2 { x: 1.0, y: 1.0 }
let v2 = v1 # 邏輯移動，物理搬運；此後 v1 不再可用
```

### 2. `ref` 與 `mut`：零拷貝借用
借用操作不會觸動所有權。在底層實作中，編譯器僅傳遞該結構的物理記憶體地址（指標）。

```valkyrie
# 僅傳遞地址，不涉及任何資料搬運
micro distance(ref a: Vec2, ref b: Vec2) -> f32 { ... }
```

### 3. 顯式複製：`.copy()` 與 `.clone()`
若需保留原物件並產生獨立副本，必須透過顯式呼叫表達：
- **`.copy()`**：執行簡單的位拷貝。僅適用於不持有複雜外部資源（如原始指標、控制代碼）的「平凡」結構。
- **`.clone()`**：觸發使用者定義的深拷貝邏輯，用於處理包含複雜權屬關係的資源複製。

## AIFD 編排：自定義行為

開發者可以透過實作特定的生命週期函式，對 `structure` 的關鍵階段實施精準干預：

- **`initiate(mut self)`**：在 A 階段（分配）之後立即觸發。常用於確立初始不變式或驗證狀態。
- **`finalize(mut self)`**：在 D 階段（釋放）之前觸發。對於 `structure`，這是**物理確定性**觸發的（即離開作用域時立即執行）。

```valkyrie
structure TempFile {
    path: utf8,
    
    # 利用 RAII 確保資源的確定性清理
    finalize(mut self) {
        # 離開作用域時，確保物理檔案被從磁碟移除
        std::fs::remove(self.path)
    }
}
```

## 硬體級物理優化 (Black Magic)

為了達成極致的執行效率，Valkyrie 編譯器在嚴格維持邏輯語義的前提下，會對 `structure` 施加多種物理層面的深度優化：

### 1. 移動消除 (Move Elision)
雖然 `move` 在邏輯上涉及拷貝，但編譯器會透過以下技術消除 90% 以上的實際開銷：
- **原地建構 (In-place Construction)**：透過 **NRVO (Named Return Value Optimization)**，編譯器直接在目標變數的物理位置上生成物件，完全跳過臨時變數的建立與搬運。
- **暫存器映射**：對於小型結構（如 `Vec2`），它們通常被直接分配到 CPU 暫存器。此時的「移動」僅是邏輯上的重命名，不產生任何記憶體 I/O。
- **尾呼叫轉發**：當變數作為最後一次使用傳遞給下游函式時，編譯器會嘗試讓新函式複用當前的記憶體 Slot。

### 2. 佈局重排 (Field Reordering)
為了對齊記憶體並減小體積，編譯器會自動優化欄位的物理順序：
- **消除空隙 (Padding)**：透過按欄位大小降序排列，擠壓掉為了記憶體對齊而產生的填充空間。
- **空間壓榨**：原本佔用 24 位元組的非對齊結構，優化後可能僅需 16 位元組，顯著提升快取行 (Cache Line) 的利用率。

### 3. 標量替換 (SROA)
若結構體僅在局部使用且未發生逃逸，編譯器會將其「拆解」為獨立的標量變數：
- **暫存器化**：結構體的各個欄位被分別映射到不同的 CPU 暫存器，完全消除記憶體訪問。
- **解除耦合**：一旦拆解，死程式碼消除等傳統優化即可針對單個欄位生效。

### 4. 零大小型別 (ZST)
不含任何欄位的 `structure` 在執行時不佔用任何物理空間 (0 Byte)。
- **零成本抽象**：你可以定義海量的 ZST 來強化型別安全或表達狀態機邏輯，而不會對生成的機器碼產生任何效能負擔。

### 5. 透明包裝 (Transparent)
單欄位包裝類別（如 `UserId(u64)`）在 ABI 層面與底層型別完全等價。
- **無感包裝**：在 FFI 呼叫中直接映射為原始型別，實作零開銷的語義強化。

### 6. 空位優化 (Niche Optimization)
編譯器會利用欄位中非法位模式（如非空指標的 0 值）來儲存 `Option` 的標籤。
- **記憶體減負**：使得 `Option<ref T>` 的大小與 `ref T` 保持完全一致，實作零空間損耗的錯誤處理。

### 7. SIMD 自動向量化
對於符合對齊要求的數值結構（如 `Vec4`），編譯器會利用 SSE/AVX 指令集實作單指令多資料並行：
- **吞吐量激增**：一次性處理 4 個或更多浮點運算，特別適用於圖形學、物理引擎及高效能科學計算場景。

## 最佳實踐：核心決策建議

- **預設首選 `structure`**：適用於數學向量、配置快照、小型狀態機以及所有追求極致效能的輕量級資料模型。
- **擁抱內聯**：透過合理組合 `structure` 減少不必要的堆積分配，構建快取友好的資料結構。
- **審視移動成本**：對於體積巨大的結構（如包含超大固定陣列），頻繁的 `own` 傳遞可能帶來物理拷貝開銷，此時建議優先考慮使用 `ref` 借用。
