# 外部物件 (Foreign Objects)

在 Valkyrie 與 C、C++ 或 Rust 等外部語言進行深度互動 (FFI) 時，嚴密且精準地管理外部物件的生命週期是確保系統穩定性、防止記憶體洩漏以及維持高效能執行的關鍵。

## FFI 與 AIFD 模型的深度映射

由於外部物件通常由其宿主語言（如 C 堆或 Rust 分配器）管理，不受 Valkyrie 垃圾回收 (GC) 機制的直接控制，開發者必須手動將其生命週期顯式映射到 AIFD 模型中。

特別是 **F (Finalize, 終結)** 階段的映射至關重要。這確保了當 Valkyrie 側的包裝物件失效時，底層的外部資源（如原始指標、系統控制代碼、網路通訊端等）能夠被確定性地釋放。

### 實戰範例：安全封裝 C++ 物件

```valkyrie
# 匯入外部 C++ 原生介面
@import(c, "lib", "obj_new")
micro obj_new() -> ◆u8

@import(c, "lib", "obj_delete")
micro obj_delete(ptr: ◆u8)

# Valkyrie 安全包裝類別
class ExternalWrapper {
    ptr: ◆u8
    
    # 狀態初始化 (I)：在建構時申請外部資源
    initiate(mut self) {
        unsafe { 
            self.ptr = obj_new() 
        }
    }
    
    # 邏輯終結 (F)：確保在物件銷毀前，外部資源被安全釋放
    finalize(mut self) {
        if self.ptr != null {
            unsafe { 
                obj_delete(self.ptr) 
                self.ptr = null # 防禦性編程：防止雙重釋放
            }
        }
    }
}
```

## 安全包裝模式 (Safe Guards)

為了提供更卓越的開發體驗 (UX) 並降低出錯風險，建議採用以下模式利用 `Finalize` 自動管理外部資源：

1.  **物件控制代碼模式 (Handle Pattern)**：將外部原始指標封裝在 Valkyrie 的 `class` 中。這樣，外部資源的存續就與 Valkyrie 物件的生命週期緊密綁定，由 GC 間接驅動清理邏輯。
2.  **RAII 守護者 (RAII Guardians)**：透過 `finalize` 方法自動觸發外部語言的解構函式或資源回收常式，實作類似於 Rust RAII 的確定性、自動化的清理機制。

## 核心記憶體安全準則

- **空指標防禦 (Null-Safety)**：在 `finalize` 的實作中，必須始終校驗外部指標的有效性。這不僅能防止雙重釋放 (Double Free)，還能增強程式在異常情況下的健壯性。
- **記憶體權屬與固定 (Pinning)**：若需要將 Valkyrie 託管的記憶體位址（如陣列緩衝區）傳遞給外部語言非同步使用，必須確保該記憶體在外部呼叫期間不會被 GC 移動。此時應使用 `Pin` 機制鎖定物件位置。

## 外部物件的底層物理優化

為了讓跨語言呼叫達到原生級別的執行效率，Valkyrie 編譯器對 FFI 互動路徑進行了深度的針對性優化：

### 1. 零複製資料交換 (Zero-copy Exchange)
Valkyrie 的 `structure` 型別在記憶體佈局上與標準 C 嚴格相容。
- **指標透傳**：在傳遞複雜結構體時，編譯器直接傳遞原始記憶體指標，徹底消除昂貴的資料複製過程。
- **常數級開銷**：即便處理包含數百萬個元素的巨型頂點緩衝區，跨語言傳遞的效能損耗也始終維持在 $O(1)$ 級別。

### 2. 極速呼叫路徑 (Fast-path FFI)
編譯器根據目標平台的 ABI（如 System V 或 Windows x64）精準編排呼叫序列。
- **暫存器直傳**：輕量級外部物件（如資源控制代碼、短向量等）直接透過 CPU 暫存器傳遞，完全繞過慢速的堆疊記憶體。
- **內建指令替換**：針對數學計算等標準庫熱點（如 `sin`, `sqrt`），編譯器會嘗試將其直接替換為等效的硬體 CPU 指令。

### 3. 終結器批次調度 (Finalizer Batching)
釋放外部物件通常涉及跨語言上下文切換的昂貴成本。
- **延遲合併策略**：GC 會智慧匯總多個待執行的外部終結邏輯，並在單次掃描週期內進行批次觸發。
- **流水線優化**：這種方式極大地減少了 CPU 在 Valkyrie 執行階段與外部執行階段之間頻繁切換產生的流水線沖刷 (Pipeline Flush) 損耗。

### 4. 包裝器剝離與內聯
- **區域性優化**：若外部包裝器實例未逃逸出當前函式作用域，編譯器會嘗試剝離包裝層，直接操作原始指標，並內聯所有的清理代碼。
- **堆疊提升**：包裝指標的 `class` 實例本身可能會被優化至堆疊上分配，從而完全消除堆分配帶來的開銷。

### 5. 跨語言全程式優化 (LTO)
當外部庫以靜態連結方式整合時，Valkyrie 支援開啟跨語言的全程式分析：
- **邊界內聯 (Boundary Inlining)**：編譯器可以打破語言壁壘，將外部庫中的熱點代碼直接內聯到 Valkyrie 的呼叫處。
- **死代碼剔除 (DCE)**：在連結階段精準識別並移除外部庫中未被呼叫的冗餘邏輯，極大地壓縮二進位體積。
