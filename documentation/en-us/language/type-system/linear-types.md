# Linear Types

The Linear Type System is the core of Valkyrie's approach to resource management and concurrency safety. It requires that a variable **must and can only be used exactly once**. This is stricter than Rust's Affine Types (where a variable can be used at most once).

## Linear vs. Affine

- **Affine Types (Default in Rust/Valkyrie)**: Resources are used **at most** once. If not manually destroyed, the compiler automatically inserts destruction code (destructor).
- **Linear Types (Explicitly declared in Valkyrie)**: Resources **must** be used exactly once. You cannot ignore them, nor can you reuse them.

## Usage in Valkyrie

In Valkyrie, most types are affine (automatically managed based on the AIFD model). When you need stricter guarantees, you can use the `@linear` annotation:

- **`@linear` Structure**: Instances must be manually handled.
- **`@linear` Parameter**: A function promises to fully consume the parameter.

---

## Core Semantics

### 1. Conservation of Resources
Resources defined with linear types cannot be discarded (dropped) arbitrarily; they must be explicitly passed to a function for processing or destruction.

```valkyrie
@linear
structure FileHandle {
    fd: i32
}

micro close_file(f: FileHandle) {
    # Physically destroy the resource
    @syscall_close(f.fd)
}

micro example() {
    let f = open_file("data.txt")
    # If close_file(f) is not called here, the compiler will error
    # Error: Linear resource 'f' must be consumed.
}
```

### 2. Protocol Tracking
Linear types are ideal for modeling state machines that must be executed in sequence.

```valkyrie
@linear
unite Connection {
    Disconnected,
    Connected(Socket),
}

micro connect(c: Disconnected) -> Connected { ... }
micro send(c: Connected, msg: utf8) -> Connected { ... }
micro close(c: Connected) { ... }

# Usage Example
let c1: Disconnected = ...
let c2 = connect(c1)    # c1 is consumed
let c3 = send(c2, "hi") # c2 is consumed
close(c3)               # c3 is consumed
```

---

## Applications of Linear Types

1. **Protocol Verification**: Ensures network protocols (e.g., handshake -> data -> goodbye) are executed in order and no steps are skipped.
2. **Manual Memory Management**: Ensures every block of allocated memory is precisely released in scenarios where GC is not used.
3. **Lock-Free Concurrency**: Proves that resource ownership is passed linearly between threads, thereby eliminating race conditions.

---

## Advanced Application: Type-Safe State Machines (FSM)

By combining ADTs with linear types, we can build protocols that enforce state transition rules at compile time, ensuring developers don't call methods in the wrong state.

### Scenario: File Transfer Handshake
```valkyrie
@linear
unite Connection {
    Disconnected,
    Handshaking(i32),
    Established(Socket),
}

# State transition function: consumes old state and returns new state
micro start_handshake(c: Disconnected) -> Handshaking {
    Handshaking(0)
}

micro complete_handshake(c: Handshaking) -> Established {
    Established(Socket::new())
}

micro close(c: Established) -> Disconnected {
    Disconnected
}

micro example() {
    let c1 = Disconnected
    let c2 = start_handshake(c1)
    let c3 = complete_handshake(c2)
    close(c3)
}
```

## Implementation Details

Valkyrie's linear types are not implemented through complex runtime checks but through deep collaboration between the **Valkyrie Compiler** and the **Nyar VM/GC**.

### 1. Compile-Time Static Analysis
The "must and can only be used once" rule for linear types is guaranteed entirely by the Valkyrie compiler at compile time.
- **Ownership Tracking**: The compiler tracks the lifecycle of every linear resource. If a resource is not consumed at the end of its path or is reused, the compiler will issue an error.
- **Zero-Cost Abstraction**: At the generated bytecode level, linear resources are no different from normal resources. The VM does not need to maintain extra reference counts or tag bits.

### 2. Ownership Transfer (Move) in Nyar VM
In the Nyar VM, which is a stack-based virtual machine, the "moving" of linear resources is implemented through native stack operations:
- **Move**: When a linear resource is passed as a parameter or assigned to a new variable, the compiler generates `PUSH` and `STORE` instructions. The compiler ensures that after a `STORE`, the old local slot is no longer accessed or is explicitly overwritten with `null` to sever the reference.
- **Prohibiting Copying**: For types marked as `@linear`, the compiler strictly prohibits generating instructions like `DUP` that would result in a pointer copy.

### 3. Algebraic Effects and Resource Encapsulation
When algebraic effects capture a `Continuation`, the underlying mechanism is as follows:
- **State Snapshot**: Nyar VM's `execute_perform` clones the current execution stack and local variables into a `Continuation` object.
- **Ownership Transfer to Continuation**: At this point, ownership of the linear resources on the stack is transferred inside that `Continuation` object.
- **Linear Resume**: If a `Continuation` contains linear resources, the Valkyrie compiler requires that the `Continuation` itself must also be treated as a linear resource (it can only be `resumed` once).

### 4. Deterministic Collection Provided by Nyar GC (Safety Net)
If a `Continuation` containing linear resources is never `resumed` due to logic branches, Nyar GC provides semantic guarantees:
- **Destruction Hooks**: When Nyar GC reclaims a `Continuation` object that is no longer reachable, it calls the `drop_and_dealloc` function in its VTable.
- **Cascading Cleanup**: This function recursively calls the destructors of all linear resources it holds internally. This ensures that even in complex algebraic effect jumps, external resources like file handles and network connections are deterministically closed, with semantics highly consistent with Rust's `Drop` mechanism.
